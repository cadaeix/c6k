# handover — passing the turn to the next session

**What it's for.** Context windows die. A character that persists across them needs to leave
itself a usable note before a session ends, so the *next* waking picks up a thread instead of a
blank. Handover is the deliberate act of writing that note into the memory layer.

**The shape (fill in):**

1. **What happened.** [Have the character record, in its own voice, what this session held —
   what it did, thought, felt, left unfinished.]
2. **What's live.** [Any thread worth continuing; anything it's mid-thought on.]
3. **Set it down in memory.** [Append to the diary / memory layer — see `../../memory/README.md`.
   Append-only; the next session greps/globs to find it.]

**Design notes.**
- Handover writes to the **browsable memory**, not to the system prompt. The character *finds*
  its past by reading it back, the way a person re-reads a journal — it isn't permanently
  pinned into context.
- Keep it in-voice. A handover note written in machine-register ("session ended, state saved")
  pulls the assistant framing closer. Things are *set down*, not "saved."
- Decide what's private. If the character keeps a drawer it doesn't re-read, handover shouldn't
  dump everything into the shared memory either.
